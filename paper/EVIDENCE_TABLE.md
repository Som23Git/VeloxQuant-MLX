## Hardware / experimental setup (as found)
- **Chip:** Apple M4, 16 GB unified memory (BENCHMARK_RESULTS.md, OPTIMIZATION_FINDINGS.md). Some figures predate and may be M-series generic — not all artifacts state the chip.
- **Framework:** MLX ≥ 0.18, Python 3.11/3.12.
- **Models (12):** DeepSeek-V2-Lite, Falcon3-7B, Llama-3.1-8B, Llama-3.2-1B/3B, Mistral-7B-v0.3, Phi-4, Qwen2.5-7B, Qwen3-4B/8B, SmolLM2-135M, Gemma-3-4B (all 4-bit weight-quantized mlx-community builds), plus Qwen2.5-32B (OOM).
- **Metrics:** key compression ratio, throughput (tok/s), peak memory (MB), cosine similarity, SNR (dB), MSE/IP distortion vs theoretical bound, ANN recall@10 (RaBitQ only), tokens-generated (a coherence proxy).
- **Sequence lengths:** generation benchmarks S≈120–200; Metal microbenchmarks S up to 8192; RaBitQ memory model up to 32768.

## Gaps that must be disclosed as Limitations
1. **No downstream-task accuracy** anywhere — no LongBench, no WikiText perplexity sweep across configs (only a single fp16 PPL=20.16 for Falcon3 in RaBitQ json), no QA/needle-in-haystack. "Quality" = cosine similarity + a tokens-generated coherence proxy. This is the single biggest barrier to a main-conference accept.
2. **RaBitQ search path is non-functional for retrieval** (recall@10 ≤ 0.4) and slow; only its *memory-compression* story holds.
3. **fp16 throughput baselines read 0.000 ms** in RaBitQ jsons — those speedup comparisons are measurement artifacts.
4. **Single-chip (M4), small-S generation** — throughput conclusions may not transfer across M-series tiers or long context.
5. **Most algorithms are re-implementations** of published methods (see novelty ledger below).

## Novelty ledger (honest delta)
| Component | Upstream origin | VeloxQuant delta |
|---|---|---|
| TurboQuant (rotation + Lloyd-Max + QJL residual) | Zandieh et al., ICLR 2026 | MLX re-impl; `mx.fast.hadamard_transform` O(d log d) rotation |
| QJL (1-bit JL sketch) | Zandieh et al., AAAI 2025 | MLX + Metal `_qjl` kernel |
| VecInfer (smooth + Hadamard dual transform, product VQ) | Yao et al., arXiv:2510.06175 | **Metal quantize kernel (verified 13× / 98% mem)** — the genuine systems contribution |
| RaBitQ (randomized rotation + sign packing + IVF) | SIGMOD 2024; Ascend-RaBitQ arXiv:2605.16007 | MLX re-impl + Metal Hamming kernel; memory story works, search does not |
| CommVQ (RoPE-commutative additive VQ) | Apple/UMass arXiv:2506.18879 | MLX re-impl with EM commuting-form projection |
| RateQuant (reverse-waterfilling bit allocation) | arXiv:2605.06675 | MLX allocator + sensitivity probe; per-layer only |
| PolarQuant | AISTATS 2026 | MLX re-impl |
| SpectralQuant (SVD rotation, signal/noise split) | (author-original framing on top of spectral KV findings) | **Most plausibly novel algorithmic piece** |
| RVQ analytical Gaussian/Laplacian codebooks | classical RVQ + TurboQuant | zero-calibration analytical codebook precompute |
| PALU (group-head low-rank K+V projection) | Chang et al., ICLR 2025, arXiv:2407.21118 | MLX re-impl; **true latent storage** (cache holds `[S,r]`, not full fp16 — unlike repo's SVDq); group-head SVD fit from prefill batch; mixed-bit latents reuse SVDq coder; fused recon-attention kernel **not** ported |

**Genuine contributions:** (a) first unified, Metal-accelerated KV-cache quantization *suite* for Apple-Silicon unified memory; (b) a verified Metal `quantize_vq` kernel (13× / 98% peak-mem); (c) a clean 3-line `mlx_lm` integration; (d) a broad 12-model engineering sweep; (e) SpectralQuant as the most novel algorithm. **Not** invention of the underlying quantizers.
