What is a KV cache?

Every time an LLM generates a token, it reads back the key and value tensors for every previous token in the context. These tensors are stored in RAM — that's the KV cache. At long contexts, it dominates memory: a 7B model at 128k tokens needs ~25 GB of KV cache alone, even on a compressed model.

What does VeloxQuant-MLX do?

It quantizes those tensors on the fly — trading a few bits of precision for dramatically less RAM. At 16× compression, a cache that needed 25 GB needs under 2 GB. Same model. More context. Runs on your Mac with near-identical output quality and unchanged mlx_lm API.

Twenty-six compression strategies

Each method targets a different point on the compression–quality–throughput tradeoff curve. All are zero-calibration and mix-and-match per layer with RateQuant. Filter by trait below, or expand the release history.

METAL-ACCELERATED IN 0.5.1
VecInfer — Product Vector Quantization
The most aggressive compression method in the library. VecInfer (Yao et al. 2025, arxiv:2510.06175) applies a dual transform to keys before product VQ encoding:

① Smooth scaling — per-channel λ = √(max|K|) suppresses outlier magnitudes
② Walsh-Hadamard rotation — spreads energy uniformly across all dims
③ K-means product VQ — encode sub-vectors against calibrated codebook

The inverse transform is absorbed into queries so q @ K.T is preserved exactly. At 1 bit/elem, a 128-dim key becomes 16 bytes instead of 256 — 16× compression.

New in 0.5.1: hand-written Metal compute kernel for quantize_vq keeps argmin in thread-local registers — 13× faster, 98% less peak memory, no API change required.

RVQ
TurboQuant RVQ
Two-pass scalar residual vector quantization. 1-bit sign quantizer + Laplacian residual stage. Achieves 7.5× compression with cosine similarity 0.92, within 5% of fp16 throughput on most 7–8B models. No calibration required.
RATE
RateQuant
Per-layer bit allocation via reverse-waterfilling (arxiv:2605.06675). 1.6s real-activation calibration measures sensitivity per layer and allocates more bits where it matters. Equal quality at lower average bit-width.
POLAR
PolarQuant
Recursive polar coordinate decomposition for clustered key distributions. Lossless angular encoding preserves inner-product structure critical to attention score accuracy.
1-BIT
QJL
Johnson-Lindenstrauss 1-bit sign sketch. An inner-product-preserving random projection for extreme compression scenarios. Minimal memory footprint with provable approximation guarantees.
KIVI
KIVI — asymmetric baseline
Re-implementation of KIVI (Liu, Yuan et al., ICML 2024, arxiv:2402.02750) — the field's most-cited KV-cache baseline. Per-channel keys, per-token values, fp16 residual window, fully deterministic (no calibration). Measured 5.8× key / ~4× full-KV at 100–106% of fp16 throughput on Llama-3.2-3B, Qwen2.5-7B, Mistral-7B (M4). Included so every other method has a recognized comparison point.
NEW IN 0.6.0
SpectralQuant — Eigenvector Rotation Quantization
The newest method in the library. SpectralQuant exploits a universal property of transformer KV caches: ~96% of key vector variance concentrates in just 3–4% of dimensions across all architectures. By rotating into the eigenvector basis before quantization, every bit lands on signal — not noise.

① Calibration — one-time SVD over 512 tokens (~5–30s) computes per-layer rotation matrices
② PCA rotation — keys are projected into their principal component basis
③ Signal/noise codebooks — separate quantizers for high-eigenvalue signal dims and low-eigenvalue noise dims
④ No QJL on noise dims — paper finding: QJL correction adds variance without reducing bias on noise dims

New in 0.6.0: calibrate_spectral_rotation(model, tokens) auto-detects architecture (including Gemma 4 multimodal), wraps sliding-window caches, and caches rotations to disk. Works with all mlx-community models.

ICML 2025
CommVQ — RoPE-Commutative VQ
Based on Apple ML Research (arXiv:2506.18879). Standard product VQ breaks with RoPE because quantize(rotate(x)) ≠ rotate(quantize(x)). CommVQ trains codebooks on pre-RoPE keys and projects each centroid onto the RoPE-commuting subspace (symmetrising paired dims), so RoPE can be applied exactly at decode time.

Result: 64× key compression (4 uint8 indices for D=128) with exact RoPE compatibility — no attention distortion from positional encoding mismatch.

RaBitQ — 1-bit Randomised Hadamard Quantization
Based on RaBitQ (SIGMOD 2024) and Ascend-RaBitQ (arXiv:2605.16007). The first method in VeloxQuant-MLX to compress both keys and values, achieving 6× total KV compression on Falcon3-7B.

① IVF clustering — K-Means partitions the key space into nList clusters
② Randomised Hadamard rotation — reuses existing mx.hadamard_transform, O(D log D)
③ 1-bit sign quantization — sign(residual) packed into D/8 uint8 bytes per key
④ Metal Hamming kernel — XOR + popcount scoring via custom rabitq_hamming_score
⑤ MSE-b4 value quantization — scalar TurboQuantMSE at 4 bits for values (4× compression)

Context capacity: at an 8 GB memory budget, fp16 fits ~17k tokens while RaBitQ+MSE fits ~103k tokens — 6× more context in the same RAM. Memory grows linearly with context length at a 6× lower slope.

NEW IN 0.16.0 — CROSS-LAYER DEPTH MERGE
MiniCache — Cross-Layer Depth-Dimension Merging
Adapted from MiniCache (arXiv:2405.14366, NeurIPS 2024). Compresses the cache across network depth: adjacent middle-to-deep layers have nearly identical KV directions, so a pair is merged into one shared SLERP direction plus each layer's own per-token magnitude — a pair of layers costs roughly one. A different route to inter-layer redundancy than XQuant: XQuant reuses codes, MiniCache merges the tensors. Zero calibration.

① Magnitude / direction split — each KV vector decomposed into length + unit direction
② SLERP merge — the two layers' directions spherically interpolated into one shared unit vector, stored once
③ Per-layer magnitudes — each layer keeps its own per-token scalar, so reconstruction is magnitude × shared_direction
④ Token retention — high-divergence token pairs (low direction cosine) are kept unmerged, capping worst-case error

Why it works: In the middle-to-deep portion of an LLM, adjacent layers' KV directions are nearly collinear — the representation evolves slowly with depth. Merging the shared direction and keeping only per-layer magnitudes recovers both layers from one stored vector.
Cross-layer mechanism SLERP tensor merge
Merged-layer cost ~1 layer per pair
Unmergeable tokens retained (cosine threshold)
Calibration None
vs XQuant merges tensors, not codes
NEW IN 0.16.0 — ENTROPY-CODED CACHE
CacheGen — Entropy-Coded KV Cache via Token Locality
Adapted from CacheGen (arXiv:2310.07240, SIGCOMM 2024). The first method in the suite to entropy-code the codes themselves rather than just pick a bit-width. Exploits token-wise locality — adjacent tokens' KV are similar — by delta-coding the quantized codes and compressing the low-entropy residual toward its Shannon entropy. Reconstruction is identical to group quant; the win is storage. Zero calibration.

① Group quantize — asymmetric min/max codes, same as KIVI
② Token-delta transform — consecutive tokens' codes subtracted; the deltas concentrate near zero
③ Entropy model — compressed size measured from the delta stream's Shannon entropy
④ Never-worse cap — bounded at fixed-width packing, so incompressible data costs 0% extra, never negative

Why it works: In long coherent contexts adjacent tokens' KV move slowly, so the per-token code deltas are low-entropy. An entropy coder spends far fewer than bits per symbol on that stream — a lossless storage win on top of the lossy quantization.
Compression axis entropy coding (lossless)
Savings on correlated KV ~10–17% over packing
Reconstruction identical to group quant
Incompressible data 0% (never negative)
Calibration None
NEW IN 0.20.0 — CONSTANT-MEMORY CACHE
StreamingLLM-adapted — Sink + Recency-Window Structural Eviction
Adapted from StreamingLLM (arXiv:2309.17453, ICLR 2024, Xiao et al.). The repo's first constant-memory cache — the first method where the cache size is bounded forever regardless of how many tokens are generated. Keeps only the first stream_n_sink token positions (frozen attention sinks) and the most recent stream_window_size tokens (rolling FIFO). All other positions are permanently evicted. Zero calibration, zero scoring — purely structural and positional.

① Freeze sinks — first stream_n_sink positions kept forever (attention sinks always receive high weight)
② Enqueue — every subsequent token enters the rolling FIFO queue
③ Trim — when FIFO exceeds stream_window_size, oldest token is evicted from the front
④ Return — concat [sink_rows || recent_rows] to the model; total positions ≤ stream_n_sink + stream_window_size

Why it works: Language models consistently attend to the initial "sink" tokens regardless of content, and attention outside the sinks is locally recency-biased. Retaining the sinks plus a recency window captures both effects at constant cost — unlike score-based eviction, which requires computing attention proxies.

Signal none — position only
Cache size constant (n_sink + window)
Works on prefill + decode (all steps)

NEW IN 0.21.0 — HEAVY-HITTER EVICTION
H2O-adapted — Cumulative Attention-Mass Heavy-Hitter Eviction
Adapted from H2O (arXiv:2306.14048, ICLR 2024, Zhang et al.). The library's third eviction axis — the first based on cumulative attention mass accumulated token-by-token during generation. On every step, each incoming key vector acts as a proxy query and attends softmax over all stored keys; the weights are accumulated into a per-token importance score. When the cache exceeds h2o_budget, the lowest-score non-sink token is permanently evicted — the cache stays at most h2o_budget positions throughout generation.

① Score — new key attends softmax over stored keys (key-as-query proxy)
② Accumulate — weights add into each token's running importance score
③ Evict — over budget → drop lowest-score non-sink token (first h2o_n_sink protected)
④ Bound — cache never exceeds h2o_budget positions

Why it works: A small set of "heavy hitter" tokens receive a disproportionate share of attention across the whole sequence. Scoring by cumulative mass keeps those tokens and drops the rest, improving over recency-only eviction (StreamingLLM) at the same constant memory cost.
Compression axis cumulative attention-mass eviction
Signal cumulative softmax mass (inertial)
Cache size constant (≤ h2o_budget)
Works on prefill + decode (all steps)

NEW IN 0.22.0 — MEMORYLESS EVICTION
TOVA-adapted — Current-Step Attention-Weight Eviction (Memoryless)
Adapted from TOVA (arXiv:2401.06104, Oren et al., 2024). The library's fourth eviction axis — and the first memoryless one: it scores tokens by the attention weight received at the current step only, carrying no history. On every step the incoming key attends softmax over the post-append cache; when over tova_budget, the lowest current-step-weight non-sink token is evicted — and the weights are then discarded. The cache stays at most tova_budget positions.

① Append — new token joins the cache
② Score now — new key attends softmax over the whole cache (current step only)
③ Evict — over budget → drop lowest current-step-weight non-sink token (first tova_n_sink protected)
④ Forget — weights discarded; nothing carries to the next step

H2O vs TOVA: H2O accumulates mass, so a past heavy hitter resists eviction (inertial). TOVA scores by the present step alone, so a token that stops being attended to is dropped even if it dominated earlier (reactive). Neither dominates — that is exactly why both are provided.

NEW IN 0.23.0 — LAYER-ADAPTIVE EVICTION
PyramidKV-adapted — Layer-Adaptive Budget Attention-Mass Eviction
Adapted from PyramidKV (arXiv:2406.02069, Cai et al., 2024). The library's fifth eviction configuration — and the first with a per-layer budget. It is H2O's cumulative-attention-mass eviction wearing a pyramid of budgets: early layers (broad attention) get a large budget, deep layers (concentrated attention) get a small one, and the average is held fixed so total memory matches uniform H2O. Flat pyramid (pyramid_beta=1.0) reduces exactly to H2O.

① Allocate — pyramid_budgets(n_layers, avg, β) gives a decreasing schedule, e.g. 12 layers @ 512 → [1019, 927, …, 97, 5] (mean 512)
② Inject — each layer's budget is baked in at for_model build time; no runtime coordinator
③ Evict — within each layer, H2O cumulative-mass scoring bounded by that layer's budget

Why a pyramid — information funneling: attention is broad in early layers and concentrates in deep ones. A uniform budget starves early layers and over-provisions deep ones; the pyramid puts the same total memory where attention actually spreads.

NEW IN 0.19.0 — TOKEN EVICTION CACHE
SnapKV-adapted — Prefill Observation-Window Token Eviction
Adapted from SnapKV (arXiv:2404.14469, ICLR 2025, Yuan et al.). The repo's first token eviction method — and the first where the paper's actual signal (attention scores) is computable at the cache level without model interception. Instead of compressing all tokens to fewer bits, SnapKV-adapted keeps only a budget of the most-attended token positions from prefill and drops the rest entirely. Zero calibration.

① Observe — last snap_obs_window key rows attend to all prefix keys via scaled softmax
② Score — mean-pool the attention matrix → [S] importance score per token
③ Select — keep the top-snap_budget tokens (plus snap_n_sink always-keep sinks)
④ Evict — drop all others permanently; retained tokens stay fp16

Why it works: Attention weight concentrates on a small fraction of prefix tokens at any generation step. Dropping the rest entirely is cheaper than quantizing them, and the observation window captures this structure from the K matrix alone — no model-forward needed.

NEW IN 0.18.0 — SALIENCY-ADAPTIVE CACHE
ZipCache-adapted — Saliency-Adaptive Per-Token Mixed Precision
Adapted from ZipCache (arXiv:2405.14256, NeurIPS 2024, He et al.). The first method in the suite on the per-token saliency axis — instead of one bit-width for every token, a small top fraction gets a higher bit-width and the rest stay at a low base rate. The paper ranks tokens by attention score; since a cache wrapper cannot observe attention scores, this implementation substitutes a key-norm proxy (the same proxy family as KIVI-Sink and AdaKV-proxy). Zero calibration.

① Score tokens — per-token key-norm proxy stands in for the paper's attention-score saliency signal
② Split by saliency — top hi_fraction of tokens flagged salient, the rest are not
③ Mixed-bit quantize — salient tokens quantized at hi_bits, the rest at lo_bits, both per-channel group quant
④ Reconstruct — dequantize each token at its assigned bit-width back to fp16 for SDPA

Why it works: Attention weight concentrates on a small set of tokens at any given step; preserving precision there while compressing the rest harder captures most of the achievable quality at a much lower average bit-width than a uniform scheme.

Saliency signal key-norm proxy (adapted)

NEW IN 0.17.0 — ERROR-FEEDBACK CACHE
GEAR — Error-Feedback KV Cache (Residual Low-Rank + Sparse Outliers)
Adapted from GEAR (arXiv:2403.05527, Kang et al.). The first method in the suite on the error-feedback axis — instead of just picking a bit-width, it reconstructs what an ultra-low-bit base quantizer threw away: X ≈ Quant_b(X) + L·R + S. Unlike CacheGen (reconstruction identical to group quant), GEAR's reconstruction genuinely recovers quality the base bit-width loses. Zero calibration.

① Base quantize — ultra-low-bit group quant, borrowed from the shared quantizer
② Residual low-rank — truncated SVD of the quantization error E = X − dequant(Quant_b(X)) gives L·R
③ Sparse outliers — the top-magnitude entries the low-rank term cannot absorb are kept exact
④ Reconstruct — base + L·R + S to fp16 for SDPA

Why it works: The residual of a coherent KV matrix is itself low-rank, so a small rank recovers most of the lost signal cheaply; a handful of sparse corrections mop up the outliers. Composes over any base quantizer to push the bit-width lower without the usual quality cliff.
Compression axis error feedback
Recovers quality at low bits
Decomposition base + low-rank + sparse
Reconstruction improved vs base quant
Calibration None
NEW IN 0.15.0 — TRUE LOW-RANK LATENT CACHE
PALU — True Low-Rank Latent Storage for Keys and Values
Adapted from PALU (arXiv:2407.21118, ICLR 2025). The first method where the cache itself stays low-rank — it stores the projected latent codes [S, r] directly and reconstructs full keys/values to fp16 only at attend time. Unlike SVDq (keys-only, reconstructs full fp16 so the win is bandwidth accounting), PALU bypasses the parent fp16 buffer entirely and compresses both keys and values — a full-KV effective rate below 1 bit/element on low-rank data. Zero calibration.

① Group-head SVD — heads are partitioned into groups; each group fits one shared projection from the prefill batch (PALU's G-LRD)
② True latent storage — the cache holds [S, r], never full [S, D] fp16; the parent ring buffer is never populated
③ Mixed-bit latents — top-25% of channels by singular value at 4-bit, the rest at 2-bit, on both K and V
④ Reconstruct on attend — L @ Vᵀ + μ rebuilds fp16 K/V for SDPA; storage stays compact

Why it works: Within a head group, keys and values share a dominant low-rank subspace — a few singular directions carry most of the attention-relevant variance. Storing the compact latent and quantizing where channel importance is already ordered by singular value beats uniform quantization in the original space, on both tensors.
Effective bit-width (K and V) ~0.6 bits (low-rank + mixed)
What the cache stores latent [S, r], not fp16
Value compression Yes — low-rank both tensors
Calibration None — SVD from prefill

NEW IN 0.14.0 — NON-UNIFORM DATATYPE
KVQuant-NUQ — Non-Uniform Quantization + Outlier Isolation
Adapted from KVQuant (arXiv:2401.18079, NeurIPS 2024). The first method in the suite to place quantization levels where the data actually is instead of uniformly. Fits non-uniform signpost levels online via Lloyd-Max and carves the few extreme outliers out to fp16 — ~73% lower reconstruction error than uniform at 3-bit on heavy-tailed K/V. Both keys and values compressed. Zero calibration.

① Lloyd-Max level fit — 2^bits signpost levels fit to the observed distribution; distortion monotone non-increasing across sweeps
② Dense/sparse isolation — top-magnitude outliers stored separately in fp16, excluded from the level fit so they cannot stretch the range
③ Per-channel keys, per-token values — key levels frozen after prefill; value levels fit per token
④ Table lookup dequant — quantize to nearest signpost index, reconstruct by gathering the level table

Why it works: LLM key/value distributions are bell-shaped and heavy-tailed. Uniform levels waste most of their range on the sparse tails; non-uniform levels concentrate precision in the dense center where the mass is, strictly reducing error at the same bit-width.
Effective key bit-width 2–4 bits (non-uniform)
Reconstruction error vs uniform ~73% lower at 3-bit

CROSS-LAYER REUSE
XQuant — Cross-Layer KV Cache Reuse
Adapted from XQuant (arXiv:2510.11236, EMNLP 2025). The first method to exploit inter-layer redundancy — adjacent layers are paired, the anchor publishes its quantized codes, and reuse layers store only their own scale/zero. Drives effective key precision below 1.4 bits/element on correlated models. Both keys and values compressed. Zero calibration.

① Anchor/reuse pairing — attention layers chunked into groups; first layer anchors, the rest reuse (group_size=2 → pairs, =3 → 1 anchor feeds 2)
② Shared coordinator — anchors publish integer codes through one cross-layer object injected at build time; mlx_lm.generate stays untouched
③ Per-layer params — reuse layers fit their own scale/zero to the shared codes, correcting cross-layer drift, storing no code tensor
④ Optional residual — a low-bit correction recovers quality when layers are less correlated

Why it works: Adjacent transformer layers produce highly similar K/V — their representations evolve gradually with depth. XQuant reuses the anchor's code assignment and pays only a fresh scale/zero per reuse layer, an axis orthogonal to every per-layer quantizer in the suite.
Effective key bit-width ~1.0–1.4 bits/element (group)
Key bandwidth reduction 11×–16× vs fp16

PER-HEAD ADAPTIVE BITS
AdaKV-proxy — Per-Head Adaptive Bit Allocation
A proxy adaptation of Ada-KV (arXiv:2407.11550), layered on KIVI-style group quantization. Allocates KV-cache bits per attention head so the average bits/element hits a configured target — high-importance heads get more bits, low-importance heads fewer. Zero calibration. Values left at fp16.

① Per-head importance — inter-token key-norm variance Var(‖k_t‖₂), an attention-free proxy for head importance, accumulated online
② Budget allocation — importances normalised and scaled to n_heads × target_avg_bits, then snapped to the allowed set {2, 3, 4}
③ Greedy round-trip correction — heads nearest a rounding boundary nudged up/down until the integer total matches the budget
④ Per-head group quantization — each head's keys quantized at its assigned bit-width, reconstructed to fp16

Why it works: Attention heads differ widely in quant sensitivity. Spending the same average budget as uniform KIVI but concentrating bits on the heads whose key magnitudes spread most across tokens recovers quality where it matters — complementary to Kitty's per-channel axis.
Effective key bit-width target (2.0–3.0), adaptive
Key bandwidth reduction 5.3×–8× vs fp16
Value compression fp16 (no change)
Calibration None — zero-shot
Decode overhead O(H) accumulator + allocation per token
ADAPTIVE MIXED-PRECISION
Kitty — Dynamic Channel-wise Mixed-Precision Keys
Inspired by Kitty (arXiv:2511.18643). Compresses keys to ~2.5 bits/element — a 6.4× memory bandwidth reduction vs fp16 — by ranking key channels online via per-channel variance and assigning 4-bit to high-variance channels, 2-bit to the rest. Zero calibration — no SVD, no codebook training. Values left at fp16.

① Online channel ranking — per-channel variance computed from the incoming key batch (prefill) or running accumulator (decode)
② Split into hi/lo sets — top hi_fraction channels (default 25%) routed to 4-bit; remaining 75% to 2-bit
③ Asymmetric group quantization — independent min/max per group along the sequence axis for each channel set
④ Running accumulator update — decode steps update key_sum and key_sq_sum in O(D) to re-rank channels

Why it works: Key channel variance is non-uniform — some channels encode far more discriminative attention patterns than others. Kitty exploits this online with no offline setup, outperforming uniform 2-bit on models with high inter-channel variance spread.
Effective key bit-width ~2.5 bits/element
Key bandwidth reduction 6.4× vs fp16
Value compression fp16 (no change)
Calibration None — zero-shot
Decode overhead O(D) accumulator update per token
NEW IN 0.10.0 — SUB-2-BIT KEYS
SVDq — Sub-2-bit Key Cache via Offline SVD
Inspired by SVDq (arXiv:2502.15304). Compresses keys to ~1.25 bits/element — a 12.8× memory bandwidth reduction vs fp16 — via a one-time SVD computed from the prefill key matrix. Values are left at fp16.

① Offline SVD at prefill — truncated SVD of K ∈ R^{S×D}, rank selected by 95% energy threshold
② Latent projection — all keys projected into [S, r] latent space via right singular vectors V
③ Mixed-precision quantization — top-25% latent channels (by singular value) at 4-bit, rest at 2-bit
④ Decode projection — each new decode key projected into the same V space, quantized, and reconstructed

Why it works: Real LLM key caches are strongly low-rank — a few singular directions carry most attention-relevant variance. SVDq exploits this by quantizing in latent space where channel importance is explicitly ordered.
Effective key bit-width ~1.25 bits/element
Key bandwidth reduction 12.8× vs fp16
Value compression fp16 (no change)

Metal Shading Language — the entire fused argmin kernel 
// One thread per sub-vector. Argmin lives in registers — no diff tensor.
uint vec_idx = thread_position_in_grid.x;
uint N_total = x_shape[0];
if (vec_idx >= N_total) { return; }

uint n_centroids = codebook_shape[0];
uint sub_dim     = codebook_shape[1];
uint x_base      = vec_idx * sub_dim;

float best_dist = INFINITY;
uint  best_idx  = 0;

for (uint c = 0; c < n_centroids; ++c) {
    uint cb_base = c * sub_dim;
    float dist = 0.0f;
    for (uint i = 0; i < sub_dim; ++i) {
        float d = float(x[x_base + i]) - float(codebook[cb_base + i]);
        dist += d * d;
    }
    if (dist < best_dist) { best_dist = dist; best_idx = c; }
}

out[vec_idx] = best_idx;
Honest caveat: the kernel pays a ~50–200µs launch overhead per call on Apple Silicon. On tiny models (SmolLM2 135M, 30 layers, ~60 launches/token) that overhead can exceed the work saved. The kernel is built for the regime that needs it — 7B+ models with realistic context lengths. Phase 2 (fused dequant + SDPA, never materializing fp16 keys) is on the roadmap. import mlx_lm
from veloxquant_mlx import KVCacheBuilder, KVCacheConfig

model, tokenizer = mlx_lm.load("mlx-community/Llama-3.1-8B-Instruct-4bit")

# 7.5× key cache compression — within 5% of fp16 throughput

import mlx_lm
from veloxquant_mlx import KVCacheConfig, KVCacheFactory
from veloxquant_mlx.allocators.vecinfer import calibrate_smooth_factors, train_codebook

model, tokenizer = mlx_lm.load("mlx-community/Qwen2.5-7B-Instruct-4bit")

# One-time calibration — run once, cache the results
smooth = calibrate_smooth_factors(sample_keys)   # [n_heads, head_dim]
codebook = train_codebook(sample_keys_flat, n_centroids=256, sub_dim=8)

# 16× key compression — Metal kernel auto-detected for 13x faster quantize
config = KVCacheConfig(
    method="vecinfer",
    head_dim=128,
    key_codebook_bits=8,      # 256 centroids
    key_sub_dim=8,             # 16× compression at 1 bit/elem
    smooth_factors=smooth,
    key_codebook=codebook,
    use_metal_kernels=None,      # None=auto, True=require, False=forbid
)
caches = KVCacheFactory.create_for_model(model, config)

response = mlx_lm.generate(
    model, tokenizer,

    max_tokens=5000,
    prompt_cache=caches,
)
python — RateQuant · per-layer mixed precision · 1.5-bit average 
from veloxquant_mlx import (
    KVCacheBuilder, KVCacheConfig,
    calibrate_layer_sensitivities,   # 1.6s one-time probe

python — SpectralQuant · 5.95× compression · best quality-per-bit · new 0.6.0 
import mlx_lm
from veloxquant_mlx.spectral.calibrate import calibrate_spectral_rotation
from veloxquant_mlx.spectral.spectral_quant import SpectralQuantizer

model, tokenizer = mlx_lm.load("mlx-community/Qwen2.5-7B-Instruct-4bit")

# Step 1 — one-time calibration (~10s). Rotations are cached to disk.
tokens = tokenizer.encode("calibration text here", return_tensors="mlx")
rotations = calibrate_spectral_rotation(
    model, tokens,
    model_name="qwen25_7b",   # cached to ~/.cache/veloxquant/
)

# Step 2 — build per-layer SpectralQuant caches
caches = []
for i, layer in enumerate(model.layers):
    key_U, _, _, _, key_ds, _ = rotations[i]
    sq = SpectralQuantizer(
        d=128, b_signal=3, b_noise=3,
        rotation=key_U, d_s=key_ds, apply_qjl=False,
    )
    caches.append(sq)  # wrap in mlx_lm-compatible cache

# Step 3 — generate as usual; 5.95× compression, +7–10pp quality vs TurboQuant
response = mlx_lm.generate(
    model, tokenizer,
    prompt="Explain quantum entanglement in simple terms.",
    max_tokens=1000,
    prompt_cache=caches,
)
